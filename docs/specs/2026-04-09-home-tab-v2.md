# Feature: Home Tab v2 — "Today's Plan" Dashboard

**Date:** 2026-04-09
**Status:** Approved
**Supersedes:** PR #26 (will be closed without merging)
**GitHub Issue:** n/a

## What

Redesign the Home tab into a single-glance daily dashboard built around the day's real events. The screen has three vertically stacked sections:

1. **Header** — time-of-day greeting ("ערב טוב, עידו · יום ג׳, 9 באפריל") + avatar
2. **Macro gauge hero** — half-circle gauge showing today's consumed-vs-target calories with a protein / carbs / fat row underneath
3. **Today's plan list** — a vertical checklist of today's meals + workout, with the next undone item highlighted in orange and an inline "Start" / "Log" pill
4. **Weekday streak strip** — 7 day-of-week dots (א-ש) with today circled + "שבוע N · X/Y אימונים" label and a flame icon

Tapping any row routes to the relevant tab (Nutrition or Workout). Rest days show a soft, non-tappable "Rest day" row in place of the workout. On fresh installs, ghost rows and a single orange "Set up your plan" CTA preview the future state. When everything is done, the orange "Next" callout collapses into a compact celebration line.

## Why

The current Home tab (PR #26's version and the earlier placeholder both) reads as an admin form: empty-state cards stacked vertically, a rotating motivational quote, and two equally-weighted quick-action buttons. Research on Hevy, MacroFactor, Fitbod, WHOOP, and Apple Fitness rings (see conversation from 2026-04-09) shows a common winning pattern: **one dominant visual metric (ring/dial) + one context-aware next action + a compact progress strip**. The mock Ido produced in Gemini fuses those into a "today's plan" checklist where the next action lives _inside_ the list rather than above it — removing the "which button do I press?" friction entirely.

Day-1 users currently see "0/3 streak · no plan yet · nutrition targets not set yet" — three separate announcements of absence. The v2 flips this to ghost rows that preview what the screen will become, plus a single directive CTA. Returning users see their calorie budget, what's next to do, and a satisfying checklist they fill in throughout the day.

## Requirements

- [ ] Greeting header shows localized time-of-day greeting + user's real name + today's date (Hebrew day-of-week + dd month)
- [ ] Macro gauge renders a half-circle arc (start at 7 o'clock, sweep to 5 o'clock) showing `consumed / target` calories in the center
- [ ] Below the gauge, a row of three columns — protein / carbs / fat — each showing `{consumed}g/{target}g` with a colored letter label (P/C/F or ח/פ/ש in Hebrew) and a colored underline
- [ ] Today's plan list is ordered chronologically: meals in `PlannedMeal.orderIndex` order, workout inserted heuristically:
  - If a `post_workout` meal exists → workout row goes immediately before it
  - Else if a `pre_workout` meal exists → workout row goes immediately after it
  - Else → workout row goes before the `dinner` meal
  - Else → workout row goes last (after all meals)
- [ ] Each meal row shows: done indicator (filled teal dot / checkmark) · meal label (Hebrew or English from i18n) · calorie target. Planned (not yet logged) meals prefix the number with `~`
- [ ] The workout row shows: done indicator · workout template name · estimated duration ("אימון דחיפה A · 45 דק׳")
- [ ] A meal row is "done" when at least one `FoodLogEntry` with a matching `mealType` and today's date exists. A workout row is "done" when `recentLogs` contains a `WorkoutLog` with today's date and non-null `completedAt`
- [ ] Exactly one row at a time is highlighted as "next" (orange glow + top-right "הבא ←" label + inline "Start" / "Log" pill button on the leading edge). The next row is the **first undone row, top-down**
- [ ] Tapping a meal row routes to the Nutrition tab. Tapping the workout row routes to the Workout tab. Tapping the rest-day row does nothing (no ripple). The inline pill button uses the same handler as its parent row

- [ ] Mixed empty state (mesocycle exists but no meal plan, or vice versa): each section independently falls back to its own empty state (gauge faded if no plan; workout row shows "קבע תוכנית אימון" CTA if no mesocycle)
- [ ] Weekday streak strip: 7 circles labeled with Hebrew day-of-week letters (א ב ג ד ה ו ש, RTL order: ש-ו-ה-ד-ג-ב-א on screen). Today's circle has a teal ring. ## Design

### Architecture

New components under `src/components/home/`:

- **`HomeHeader.tsx`** — greeting + avatar + date row
- **`MacroGauge.tsx`** — half-circle SVG gauge (new — `MacroRing` stays for Nutrition tab)
- **`MacroLegend.tsx`** — P/C/F three-column row (consumed/goal per macro)

New pure function under `src/utils/`:

- **`buildTodaysPlan.ts`** — takes `(activePlan, todaysWorkoutDay, todaysLog, today)` → returns an ordered array of `PlanItem` with `{kind: 'meal' | 'workout' | 'rest', id, label, calories, duration?, done, isGhost}`. Pure, easy to unit-test. This is where the workout-insertion heuristic lives.

Modified:

- **`app/(tabs)/index.tsx`** — swaps placeholder header for `HomeHeader`, then renders `MacroGauge` + `MacroLegend` + `TodaysPlanList` + `WeekdayStreakStrip` inside a `ScrollView`

- **`src/stores/useUserStore.ts`** — (read-only) expose user's first name for the greeting (already stored during onboarding)

Unchanged (reused):

- **`Card`** — for section containers
- **`RTLWrapper`** — inside every row + header
- **`useWorkoutStore`** selectors: `plan`, `mesocycle`, `getTodaysWorkout`, `getCompletedThisWeek`
- **`useNutritionStore`** state: `activePlan`, `todaysLog`, `dailySummary`

Deleted:

- **`src/components/home/TodaysWorkoutCard.tsx`** + test (replaced by `PlanRow` + `buildTodaysPlan`)
- **`src/components/home/TodaysMacrosCard.tsx`** + test (replaced by `MacroGauge` + `MacroLegend`)

### Data Flow

```
useWorkoutStore.plan + getTodaysWorkout
useNutritionStore.activePlan + todaysLog + dailySummary
                ↓
        buildTodaysPlan() (pure)
                ↓
        PlanItem[] (ordered, with done flags, with `isNext` marker for first undone)
                ↓
        TodaysPlanList renders each as PlanRow
                ↓
User taps row → useRouter().push('/(tabs)/nutrition' | '/(tabs)/workout')
                ↓
        On return to Home, store state has changed → recompute plan → done flags flip
```

**Shared type — define in Task 1, used by every downstream task:**

```ts
// src/utils/buildTodaysPlan.ts
export type PlanItemKind = 'meal' | 'workout' | 'rest' | 'ghost'

export interface PlanItem {
  readonly kind: PlanItemKind
  readonly id: string // stable key: mealType for meals, 'workout' / 'rest' / 'ghost-N'
  readonly labelKey: string // i18n key under home.v2.planItems (e.g. 'breakfast')
  readonly calories: number | null // null for workout + rest rows
  readonly durationMinutes: number | null // set only for workout rows
  readonly done: boolean // false for rest + ghost
  readonly isNext: boolean // true only for the first undone, non-rest, non-ghost row
  readonly routeTarget: 'nutrition' | 'workout' | 'onboarding' | null
}

export interface BuildTodaysPlanInput {
  readonly activePlan: MealPlan | null
  readonly todaysWorkoutDay: GeneratedWorkoutDay | null
  readonly todaysLog: readonly FoodLogEntry[]
  readonly todaysCompletedWorkoutLog: WorkoutLog | null
  readonly mealsPerDayFallback: MealsPerDay // used only in ghost mode
}
```

The `isNext` flag is computed by `buildTodaysPlan` itself so `TodaysPlanList` stays a dumb renderer. This is the key invariant: **the pure function owns all layout logic; the component just maps PlanItem[] → JSX.**

### Task 1: Define `PlanItem` types + `buildTodaysPlan` test scaffolding (M)

**Files:**

- Create `src/utils/buildTodaysPlan.ts` with **only the type exports** (no function body yet)
- Create `src/utils/buildTodaysPlan.test.ts` with ALL failing test cases (RED)

**What:** Lay down the type contract for the pure function and write every test case before touching the implementation. Tests cover:

1. **Ordering rules** (4 cases — one per heuristic branch):
   - Plan has a `post_workout` meal → workout row appears immediately before it
   - Plan has only a `pre_workout` meal → workout row appears immediately after it
   - Plan has neither, but has a `dinner` meal → workout row appears immediately before dinner
   - Plan has none of the above → workout row appears last
2. **Done detection**:
   - Meal is done when ≥1 `FoodLogEntry` with matching `mealType` exists for today
   - Meal is pending when the log has entries for _other_ meal types only
   - Workout is done when `todaysCompletedWorkoutLog` is non-null and has `completedAt !== null`
3. **`isNext` marker**:
   - Exactly one row has `isNext: true` at any time (the first undone row, top-down, skipping rest + ghost rows)
   - When all rows are done → zero rows have `isNext: true` (celebration state)
   - Rest-day row is never `isNext`
   - Ghost rows in fresh-install mode: the FIRST ghost row gets `isNext: true`
4. **Rest day**:
   - When `todaysWorkoutDay.template === null`, a `{kind: 'rest'}` row replaces the workout row at the same position
5. **Ghost / fresh-install mode**:
   - When both `activePlan === null` and `todaysWorkoutDay === null` → returns 5 ghost rows (5 is from `mealsPerDayFallback`: 4 meals + 1 workout) with the first having `isNext: true`
   - Ghost rows have `done: false`, `calories: null`, `routeTarget: 'onboarding'`
6. **Mixed empty state**:
   - `activePlan === null` but workout day exists → ghost meal rows + real workout row
   - Workout day is null but `activePlan` exists → real meal rows + no workout row (or a "set up training" ghost workout row — decide in test: preferred "no workout row at all" for v1 simplicity)
7. **Pure-function guarantees**:
   - Same input → same output (no `Date.now` inside; function takes a `today: string` parameter passed in)
   - Input arrays are not mutated

**Test first:** All of the above, written as `it(...)` blocks with explicit expected arrays. Use realistic fixture builders: `makePlan()`, `makeWorkoutDay()`, `makeLogEntry()`.

**Acceptance:** `npx jest src/utils/buildTodaysPlan.test.ts` reports "Cannot find module" or "not a function" — confirming all tests fail for the right reason. Then the file compiles with just the type exports, leaving the test suite RED because the function body is missing.

### Task 2: Implement `buildTodaysPlan` (GREEN) (M)

**Files:**

- `src/utils/buildTodaysPlan.ts` — add the function body

**What:** Make every test from Task 1 pass. Purely functional: reads only from its arguments, uses `Set` / `Map` / `Array` methods only, never touches stores or `Date` globally. The `today` parameter is an ISO date string (`'2026-04-09'`) passed in from the caller.

Algorithm outline:

1. If fresh-install (`activePlan === null && todaysWorkoutDay === null`) → synthesize 4 ghost meal rows + 1 ghost workout row → mark first as `isNext` → return.
2. Otherwise, assemble the real rows:
   - Map `activePlan.days[today.dayOfWeek].meals` → meal rows in `orderIndex` order
   - Apply the workout-insertion heuristic (`post_workout` → `pre_workout` → `dinner` → end)
   - If workout day exists but `template === null` → insert a rest row instead
   - Compute `done` for each meal from `todaysLog`, and `done` for the workout from `todaysCompletedWorkoutLog`
3. Compute `isNext`: find the first row where `done === false && kind !== 'rest' && kind !== 'ghost'` — set `isNext: true` on that row only.
4. Return the array.

**Test first:** (written in Task 1, already RED)

**Acceptance:** `npx jest src/utils/buildTodaysPlan.test.ts` → all tests GREEN. `npm run typecheck` clean.
